# Multi-Cloud IAM Least Privilege Framework

A production-grade IAM security framework across **AWS**, **GCP**, and **Azure** using Terraform.

Implements least-privilege policies, credential-free Kubernetes workloads via Workload Identity, and automated daily security auditing across all 3 clouds.

> **Real finding:** After building this framework I ran the audit scripts against my own accounts and found **6 HIGH severity issues** — including wildcard policies auto-generated by AWS CodePipeline and a GCP default service account with `roles/editor`. Details in the [audit results](#audit-results) section.

## Key Concepts

### Workload Identity — No Static Credentials in Pods

The most important security pattern in this project. Pods access cloud services without storing any credentials.

**Old way (wrong):**
```yaml
env:
  - name: AWS_ACCESS_KEY_ID
    value: "AKIAIOSFODNN7EXAMPLE"      # static key in pod — security risk
  - name: AWS_SECRET_ACCESS_KEY
    value: "wJalrXUtnFEMI/K7MDENG..."
```

**New way (right):**
```yaml
# AWS — annotate the ServiceAccount with the IAM role ARN
apiVersion: v1
kind: ServiceAccount
metadata:
  name: app-sa
  namespace: app
  annotations:
    eks.amazonaws.com/role-arn: arn:aws:iam::ACCOUNT_ID:role/roles/dev-app-irsa-role
```

The pod gets short-lived tokens automatically. No credentials to store, rotate, or leak.

| Cloud | Mechanism | K8s Annotation Key |
|---|---|---|
| AWS EKS | IRSA via OIDC | `eks.amazonaws.com/role-arn` |
| GCP GKE | Workload Identity | `iam.gke.io/gcp-service-account` |
| Azure AKS | Federated Credentials | `azure.workload.identity/client-id` |

### Least Privilege Policies

Every policy is scoped to exact resources — no wildcards.

```hcl
# Good — scoped to one S3 bucket and one secret prefix
Statement = [
  {
    Effect   = "Allow"
    Action   = ["s3:GetObject", "s3:ListBucket"]
    Resource = ["arn:aws:s3:::my-app-bucket", "arn:aws:s3:::my-app-bucket/*"]
  },
  {
    Effect   = "Allow"
    Action   = ["secretsmanager:GetSecretValue"]
    Resource = "arn:aws:secretsmanager:us-east-1:ACCOUNT:secret:dev/app/*"
  }
]
```

```hcl
# GCP — custom role with only 4 permissions instead of roles/editor
permissions = [
  "storage.objects.get",
  "storage.objects.list",
  "pubsub.topics.publish",
  "secretmanager.versions.access",
]
```

## Getting Started

### Prerequisites

- Terraform >= 1.5
- AWS CLI configured (`aws configure`)
- GCP CLI configured (`gcloud auth application-default login`)
- Azure CLI configured (`az login`)
- kubectl
- Python 3 + boto3 (`pip install boto3`)

### AWS Deployment

```bash
cd terraform/aws

cp terraform.tfvars.example terraform.tfvars
# Edit terraform.tfvars with your account_id, region, etc.

terraform init
terraform plan
terraform apply
```

After apply, test IRSA:

```bash
# Get kubectl credentials
aws eks update-kubeconfig --name dev-eks-cluster --region us-east-1

# Deploy and verify
bash k8s/verify-irsa.sh
```

Expected output:

```json
{
  "Arn": "arn:aws:sts::ACCOUNT_ID:assumed-role/dev-app-irsa-role/...",
  "UserId": "AROAVRUVVS7YF26Q4GO76:...",
  "Account": "ACCOUNT_ID"
}
```

### GCP Deployment

```bash
cd terraform/gcp

# Edit terraform.tfvars with your project_id
terraform init
terraform plan
terraform apply

# Get kubectl credentials
gcloud container clusters get-credentials dev-gke-cluster --zone us-central1-a

# Deploy and verify
bash k8s/gcp/verify-workload-identity.sh
```

### Azure Deployment

```bash
cd terraform/azure

# Edit terraform.tfvars with your subscription_id and tenant_id
terraform init
terraform plan
terraform apply

# Get the managed identity client ID
terraform output app_identity_client_id

# Update the client ID in the K8s manifest
sed -i 's/REPLACE_WITH_CLIENT_ID/<client_id>/' k8s/azure/workload-identity-serviceaccount.yaml

# Get kubectl credentials and verify
az aks get-credentials --resource-group dev-iam-project-rg --name dev-aks-cluster
bash k8s/azure/verify-workload-identity.sh
```

Expected output:

```json
{
  "user": {
    "name": "d7bf11bb-5513-445f-9869-de5b09e13c03",
    "type": "servicePrincipal"    ← Managed Identity, not personal account
  }
}
```

## Running the Audit Scripts

```bash
# Install Python dependency
pip install boto3

# Run individual audits
python3 audit/aws_iam_audit.py
python3 audit/gcp_iam_audit.py
python3 audit/azure_rbac_audit.py

# Run all 3 at once
bash audit/run_all_audits.sh
```

The scripts exit with code `1` if any HIGH severity findings exist — making them compatible with CI/CD pipelines.

## Lessons Learned

**1. Never trust auto-generated IAM**
AWS CodePipeline, Elastic Beanstalk, and several other services create IAM roles with wildcard permissions automatically. Always audit them.

**2. GCP starts permissive by default**
The default compute service account gets `roles/editor` on every new project. Remove it immediately.

**3. OIDC thumbprint can mismatch**
If Terraform fetches the thumbprint while the EKS cluster is still initializing, it may get the wrong intermediate certificate. Fix with:
```bash
aws iam update-open-id-connect-provider-thumbprint \
  --open-id-connect-provider-arn "arn:aws:iam::ACCOUNT:oidc-provider/..." \
  --thumbprint-list "CORRECT_THUMBPRINT"
```

**4. IAM role path affects the ARN**
A role created with `path = "/roles/"` has ARN `arn:aws:iam::ACCOUNT:role/roles/role-name` — not `role/role-name`. Always use `terraform output` to get the exact ARN instead of constructing it manually.
